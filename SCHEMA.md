# UX4 Flows Schema

Canonical behavioral spec schema for conversational agents authored in UX4. The UX4 editor (this repo, `uxflows`) is the primary producer. UX4 simulation, evaluation, and client-facing document rendering (sibling repo `../whatsupp2/`) are the primary consumers.

**This document is the authoritative schema.** Both repos defer to it. Schema-as-code (TypeBox) mirrors this document at `lib/schema/` (not yet built).

## Status

Two versions:

- **v0** — the MVP schema. Documented in full below ([Agent Schema](#agent-schema), [Flow Schema](#flow-schema)).
- **v1** — forward-looking. Adds step-level `tool` and `call` types and a flow-level `pipecat` hints field. Additive: every valid v0 spec is a valid v1 spec. Documented in [Schema v1 (Forward-Looking)](#schema-v1-forward-looking).

## Conventions

### Variable Naming

Lowercase with underscores. No camelCase. No hyphens. No spaces. Variable names become Python identifiers and JavaScript object keys in generated code across all export targets.

Type declarations are preferred for variables used in calculation expressions or function definitions. When a variable has no type declaration, the translator defaults to string.

### IDs

Anything commentable or referenceable has a stable `id`. IDs are generated by the editor, never authored manually. The cost is negligible; future flexibility (comments, references, diffing) is real.

### Optional By Default

The schema skeleton is valid with minimal required fields. Depth is added incrementally as the author builds confidence.

### Schema vs UI

Schema defines behavior. UI defines rendering. Node position on the canvas, color coding, panel state are UI concerns and must not appear in exported spec JSON.

### Execution Separate From Spec

Endpoint, headers, and runtime model live in a separate `execution` object outside the spec. Sharing or exporting a spec must not leak credentials or deployment details.

`system_prompt` and `chatbot_initiates` live *inside* the spec because they describe agent behavior, not hosting.

## Three Methods

Three methods apply uniformly wherever a value is determined or a condition is checked — in `captures`, `conditions`, `assigns`, and `entry_conditions`.

- **`llm`** — Semantic understanding. The LLM reads conversation content and judges or extracts. Probabilistic. Subject to variance. Used when the answer requires understanding meaning.
- **`calculation`** — Deterministic computation. Evaluates a Python-like expression over known variable values. Pattern matching over string structure (for IVR or legacy NLU) is a subtype, represented with `method: "calculation"` and an optional `pattern` field.
- **`direct`** — A literal value. No computation. No LLM.

### Calculation Expression Syntax

Python-like expressions that translate directly into generated code without parsing ambiguity.

- Comparison: `==`, `!=`, `>`, `>=`, `<`, `<=`
- Logical: `and`, `or`, `not`
- Literals: strings in double quotes, numbers unquoted, booleans as `True` / `False`, `None` for null.

Examples:

```
verified == True
loan_amount > 50000
days_overdue >= 30
loan_amount > 50000 and verified == True
status == "overdue"
payment_received == None
```

LLM conditions remain plain-language strings and are never emitted into generated code. The `method` field determines which interpretation applies.

## Variable Scope

Variables exist at two levels.

**Agent-level** — declared in the agent's `variables` dictionary. Available to all flows. Each variable has a `scope`:

- `plan_level` — same for all users in a deployment (agent name, company name, fee percentage, support phone number). Set once per deployment.
- `per_persona` — specific to the individual being contacted (customer name, loan amount, due date, account details). Provided per conversation, typically by the runtime before the conversation begins.

**Flow-level** — local to a flow unless produced through `exit_paths[].assigns`. A flow variable with the same name as an agent variable inherits type and example but can be overridden within the flow.

## Agent Schema

```json
{
  "$schema": "UX4://agent/v0",
  "id": "string",
  "version": "string",

  "meta": {
    "name": "string",
    "purpose": "string",
    "client": "string",
    "language": "string",
    "user_segments": "string (freeform description of who the agent talks to)"
  },

  "system_prompt": "string",
  "chatbot_initiates": "boolean",

  "variables": {
    "variable_name": {
      "type": "string | number | boolean | enum",
      "scope": "plan_level | per_persona",
      "description": "string",
      "example": "any"
    }
  },

  "guardrails": [
    { "id": "string", "statement": "string" }
  ],

  "knowledge": {
    "faq": [
      {
        "id": "string",
        "question": "string",
        "answer": "string",
        "scripts": { "EN": "string", "ES": "string" }
      }
    ],
    "glossary": [
      { "id": "string", "term": "string", "definition": "string" }
    ],
    "sources": [
      {
        "id": "string",
        "name": "string",
        "description": "string",
        "type": "rag | tool | api"
      }
    ],
    "tables": [
      {
        "id": "string",
        "name": "string",
        "purpose": "string",
        "structure": [
          { "field": "string", "description": "string", "type": "string" }
        ],
        "rows": [
          { "field_name": "value" }
        ],
        "scaling_rule": "string (optional)"
      }
    ]
  },

  "entry_flow_id": "id of the flow in the top-level spec.flows[] that the conversation starts with"
}
```

### Field Notes

- **`meta`** — what the agent is and who it serves. Appears at the top of the shared client document. Includes `user_segments`, a freeform description of who the agent talks to; behaviorally informative (the agent's tone, vocabulary, and assumptions should fit the target audience) and also consumed downstream in `../whatsupp2/` as a seed for persona generation (persona definitions themselves live there, not in the spec).
- **`system_prompt`** — the agent's behavioral instructions as authored. Present when flows are not yet defined or when the prompt carries behavioral intent not yet decomposed into flows. Part of the spec because it describes behavior, not hosting.
- **`chatbot_initiates`** — behavior flag: whether the agent sends the first message or waits for the user.
- **`variables`** — agent-level variable declarations. Variable name is the key and stable identifier. Authoritative source of type, scope, description, and example for variables shared across flows.
- **`guardrails`** — cross-cutting behavioral invariants. Evaluated against the full transcript. Each guardrail is a stable `id` and a single `statement` sentence. Conditional rules are written inline as natural language ("If X, always Y"). Executable conditional routing belongs in interrupt flows, not guardrails.
- **`knowledge.faq`** — high-confidence answers authored with the client. Optional per-language scripts capture the actual phrasing in each supported language.
- **`knowledge.glossary`** — domain terms ensuring consistent terminology.
- **`knowledge.sources`** — external capabilities declared by name, description, and optional type. The description tells the agent when to use this source.
- **`knowledge.tables`** — structured lookup data (promo plans, payment tier thresholds, branch hours, fee schedules). Each has a `purpose`, a `structure` (field definitions), sample `rows`, and an optional `scaling_rule` describing how values adapt to real inputs.
- **`entry_flow_id`** — points at one of the flows in the top-level `spec.flows[]`. The flow the conversation enters first.

## Flow Schema

```json
{
  "$schema": "UX4://flow/v0",
  "id": "string",
  "version": "string",
  "name": "string",
  "description": "string",
  "type": "happy | sad | off | utility | interrupt",
  "scope": ["global"] | ["flow_id_1", "flow_id_2"],

  "variables": {
    "variable_name": {
      "type": "string | number | boolean | enum",
      "scope": "plan_level | per_persona",
      "example": "any"
    }
  },

  "steps": [
    {
      "id": "string",
      "type": "turn",
      "role": "agent | user",
      "label": "string",
      "content": "string",
      "variables_used": ["variable_name"],
      "condition": {
        "id": "string",
        "expression": "string",
        "method": "llm | calculation | direct"
      },
      "captures": [
        {
          "id": "string",
          "variable": "string",
          "type": "string | number | boolean | enum",
          "method": "llm | calculation | direct",
          "pattern": "string (optional, when method is calculation)",
          "value": "any (used when method is direct)"
        }
      ],
      "utterances": [
        {
          "id": "string",
          "language": "string",
          "dialect": "string",
          "variations": [
            { "id": "string", "text": "string" }
          ]
        }
      ]
    }
  ],

  "success_criteria": [
    { "id": "string", "criterion": "string" }
  ],

  "guardrails": [
    { "id": "string", "statement": "string" }
  ],

  "max_turns": "number (optional; flow follows unconditional sad exit_path on exhaustion)",

  "example": "string (plain-text transcript; free-form format)",

  "routing": {
    "entry_conditions": [
      {
        "id": "string",
        "expression": "string",
        "method": "llm | calculation | direct"
      }
    ],
    "exit_paths": [
      {
        "id": "string",
        "type": "happy | sad | off | exit | return_to_caller",
        "condition": {
          "id": "string",
          "expression": "string",
          "method": "llm | calculation | direct"
        },
        "next_flow_id": "string | null",
        "assigns": {
          "variable_name": {
            "method": "llm | calculation | direct",
            "value": "any"
          }
        }
      }
    ]
  }
}
```

### Field Notes

- **`type`** — labels the flow for the canvas and for evaluation. `happy` = success path. `sad` = failure path. `off` = off-topic. `utility` = shared infrastructure (verification, disclosure). `interrupt` = fires outside normal sequence in response to triggers.
- **`scope`** — meaningful for `interrupt` flows. `["global"]` allows the interrupt to fire from any flow. A list of flow IDs scopes the interrupt to specific flows. Omitted for non-interrupt flows.
- **`steps`** — ordered. Each step has a stable ID. v0 only supports `turn` steps.
- **Turn `condition`** — optional. When present, determines whether this turn is active given current conversation state.
- **Turn `captures`** — extract values from the conversation. Meaningful on user turns. `llm` extracts from natural language. `calculation` derives from other variables or pattern-matches against the user's message. `direct` assigns a literal.
- **Turn `utterances`** — actual script content per language and dialect. Variations within a language are equivalent alternatives. Agent utterances are authoritative — the agent says one of these. User utterances are illustrative — examples of what a user expressing this content might say.
- **Symmetric turns** — agent and user turns share the same structure. Role determines interpretation.
- **`routing.entry_conditions`** — when this flow should be active. For interrupt flows these are trigger phrases or intents. For sequential flows these are the conditions under which the routing layer enters the flow.
- **`routing.exit_paths`** — how the flow ends. `next_flow_id` is the flow to enter next. `return_to_caller` is the special exit for interrupt flows, resuming whichever flow was interrupted. `assigns` produces variables that become available to the next flow as agent-level variables.
- **`max_turns`** — optional integer. When set, the flow follows its unconditional sad `exit_path` (the "give up" branch) after user+agent turn pairs in this flow reach this count. Interrupts don't count. If no unconditional sad exit_path exists, the flow ends. Convention over configuration — no paired exit_path_id field.
- **`example`** — optional plain-text transcript illustrating intended behavior for this flow. Free-form format (the author chooses the shape — e.g., `Agent: ...\nUser: ...`). Annotation-only: runtimes ignore it; simulation may consume it as a seeding hint but does not assert against it. Illustrative, not authoritative; the authoritative per-turn agent scripts live in `steps[].utterances`.

## Common Flow Compositions

- **Conversational flow** — only turn steps. The default.
- **Deterministic call tree** — turn steps with calculation conditions and direct or pattern captures. No LLM judgment in routing or capture. Compatible with IVR and Twilio ConversationRelay DTMF handling.
- **Free-form node** — a flow with no turns. The LLM handles conversation freely within the flow-level guardrails and success criteria.
- **Interrupt flow** — `type: interrupt` with `scope: ["global"]` or a list of flow IDs. Entry conditions match trigger phrases or intents. Exit paths use `return_to_caller` to resume the interrupted flow.
- **Tool-augmented / mixed / sub-flow orchestrator** — require v1 step types (`tool`, `call`). Not supported in v0.

## What v0 Does Not Include

- `tool` step type (external capability calls — MCP, RAG, API) — added in v1.
- `call` step type (sub-flow invocation with input/output mapping) — added in v1.
- `pipecat` hints field at the flow level (runtime hints with no behavioral equivalent) — added in v1.

v0 schemas remain valid v1 schemas. v1 schemas with additional features cannot be downgraded to v0 without information loss.

## Schema v1 (Forward-Looking)

v1 extends v0 with step-level structure that supports tool calls and sub-flow invocation, plus a flow-level field for Pipecat-specific runtime hints. v1 additions are purely additive at the schema level — no v0 spec needs restructuring to become a v1 spec.

### v1 Step Types

v0 supports only `turn` steps. v1 adds two additional step types.

#### Tool Step

```json
{
  "id": "string",
  "type": "tool",
  "description": "string",
  "tool_type": "mcp | rag | api",
  "inputs": ["variable_name"],
  "outputs": ["variable_name"],
  "connection": "string",
  "parameters": {}
}
```

A non-conversational step that calls an external capability — MCP server, RAG retrieval, API call. Takes input variables. Produces output variables. The `description` declares what this step does in plain language. Not visible in the client document. Variable types should be declared for all tool inputs to enable clean function-definition generation across all export targets.

#### Call Step

```json
{
  "id": "string",
  "type": "call",
  "description": "string",
  "target_flow_id": "string",
  "target_flow_version": "string",
  "input_mapping": {
    "agent_variable": "flow_consumed_variable"
  },
  "output_mapping": {
    "flow_produced_variable": "agent_variable"
  }
}
```

Invokes another flow inline and returns. The called flow runs completely. Produced variables are returned through `output_mapping`. Execution continues in the parent flow after the called flow completes. Not visible in the client document.

### v1 Flow-Level Additions

v1 adds an optional `pipecat` hints field at the flow level:

```json
{
  "pipecat": {
    "context_strategy": "APPEND | RESET",
    "respond_immediately": true,
    "pre_actions": [],
    "post_actions": []
  }
}
```

Pipecat-specific configuration with no behavioral-spec equivalent. Passes through verbatim to the generated Pipecat node configuration. Behavioral fields are never included here. Lives at the flow level specifically to keep runtime hints out of the behavioral spec layer.

### v1 Variable Scope and Mapping

Variables exist at two levels in v0 — agent and flow. v1 adds explicit cross-flow variable mapping through call steps.

When a flow is invoked by a call step:

- `input_mapping` maps agent variables to the flow's consumed variables.
- `output_mapping` maps the flow's produced variables back to agent variables.

Variables do not leak across flow boundaries without explicit mapping.

Produced variables from `exit_paths[].assigns` become available to the next flow in the routing chain as agent-level variables.

Within a flow, variables referenced in step content are *consumed*. Variables assigned in exit paths are *produced*. Direction is derived, not declared.
